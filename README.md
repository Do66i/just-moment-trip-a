🙋‍♂️   **Redux가 아닌 HOOK을 사용하여 상태관리를 한 이유는 무엇인가요 ?** 

🥔  Diary 및 Account의 로직을 구현할 때 `useReducer`라는 HOOK을 이용한 이유는 상태변화 로직을 컴포넌트화 한 후 분리하여 타 컴포넌트들의 코드가독성을 높히고 컴포넌트의 무게를 가볍게 하기 위함입니다. 프론트엔드를 맡으신 또 다른 팀원분은 Redux를 사용하였지만 저는 Redux 대신 React의 HOOK을 선택한 이유는 순수 리엑트 자체만으로 상태를 관리하는 것에 좀 더 집중(도전)하고 싶었고, 이번 프로젝트를 시작하면서 우리 팀은 **기본적인 구현을 완벽하게 하자**라는 목표를 세웠고 그 목표를 달성하기 위해서 저는 새 스택을 사용하지 않는 것을 선택하게 되었기 때문입니다.

---

🙋‍♂️  **어째서 상태관리를 하는 컴포넌트(DiaryStore.js & AccountStore.js) `useCallback` 를 사용하였나요 ?**

🥔  컴포넌트를 작성하며 마주친 렌더링 이슈로 먼저 `useEffect`를 사용하였습니다. 하지만 이 HOOK을 사용하여도 렌더링 이슈가 발생하였습니다. 이슈의 발생 원인을 찾아보니 함수(비원시적인 타입)의 재생성으로 인한 이슈라는 것을 알게 되었습니다. 이 문제를 해결하기 위해 구글링, 유튜브 검색 및 React 공식문서를 통하여 `useCallback`이라는 HOOK을 알게 되었는데, 이 HOOK은 두번째 인자로 받은 배열의 변경사항이 없다면 메모이제이션 된 값을 반환(콜백함수)한다는 점을 알게 되어 렌더링 이슈를 해결하기 위해 되어 사용하였습니다.

🙋‍♂️  **프로젝트 진행 중 가장 힘들었던 점은 무엇인가요 ?** 

🥔  역시 멘탈적인 부분(팀원들과 월등하게 차이나는 실망스러운 내 실력)인 것 같습니다. 팀원들이 이루고자 하는 것을 위해 밤을 새거나 식사시간도 줄여가며 코드작업을 하는것은 괜찮았으나(오히려 팀원들과 함께하며 배울것들이 많아서 더 즐거웠고 좋았습니다), 백엔드분들은 실력이 월등한데 내가 실력이 부족해서 그분들이 구현하고자 하는것들은 나때문에 진행하지못하거나 내 진행상황이 늦어져서 팀원들이 나 때문에 계획을 바꾸게 되는 상황을 접할때마다 마음이 너무 힘들었습니다. 지난 프로젝트때는 혼자만의 문제라 생각하고 공유하지 않았으나 이번 프로젝트때는 그런 부정적인 생각이 들 때 마다 팀원들에게 제 상황을 공유를 하였고 팀원들의 의견 및 격려를 들으며 멘탈을 부여잡을 수 있었고 개인적으로 주말이나 점심시간에 강도가 높은 운동을하거나  DevLog나 KPT회고 그리고 블로그 작성을 통해 제 속마음을 정리하고 다독여가며 멘탈을 관리하였습니다.

🙋‍♂️  **가장 기억에 남는 에러핸들링이 있나요 ?**

🥔  해시태그 구현을 할 때, 데이터 값을 꼼꼼하게 확인하지 않아서 만나게 된 에러가 가장 기억에 남습니다. 객체로 들어오는 data값(이하 state)을 당연히 배열로 들어온다는 생각을 하여 `map()` 를 사용하여 해시태그를 보여지게 하려고 하였으나 어찌된 일인지 `state.map is not a function` 이라는 에러를 console창에서 보게 되었습니다. state의 값을 확인 할 생각을 못한 채 단순히 제가 값을 잘못 받아왔거나 메소드를 사용하는 순서를 잘못 작성한 줄 알고 엄한 시간을 소비했었습니다. 결국 백엔드 팀원분들의 도움으로 에러를 해결하게 되었습니다. 수 많은 에러들 중에 이 에러가 제 기억에 잔상으로 진하게 남은 이유는 로직을 짜기 전 꼼꼼하게 확인하는 작업을 가지게 된다면 아까운 시간을 소비하지 않을 수 있다는 생각을 하게 되었고, 코드를 작성하기 전 먼저 사용하려고 하는 data값의 타입을 주석으로 작성하고 `console.log()`를 사용하여 항상 확인 할 수 있게 설정하는 습관을 가지게 됐기 때문입니다.

    Q. cron을 왜 쓰셨나요?
    
    A. 현재 DB스키마의 형태가 
    
    user : trip = 1 : N
    
    trip : diary = 1 : N
    
    diary : hashtag  = N : M 입니다.
    
    sequelize를 이용한 관계설정을 해줬기때문에 상위 테이블인
    
    user 삭제시 자동으로 하위 테이블인 trip, diary, diary_hashtag join table이 삭제됩니다.
    
    그러나 hashtag까지 삭제되지않는데 그 이유는 diary와 hashtag는 N : M 연결이기 때문입니다.
    
    즉, 특정 user의 hashtag가 삭제되더라도 다른 user가 해당 hashtag를 쓰고 있을 수 있기 때문입니다.
    
    그런데 join table에서 참조하지 않는 hashtag값이 존재할시에 그 값은 아무곳에서도 참조하지않는 쓰레기값이 됩니다. 즉, 메모리 낭비가됩니다.
    
    그렇다고 delete나 update요청시마다 쓰레기값여부를 확인한다면 리소스 낭비이므로 cron을 이용한 스케쥴러를 등록해서 주기적으로 쓰레기값을 삭제해주었습니다.

Q : 왜 HTTPS로 배포하였나요

A : 저희는 회원가입을 요구하고 로그인을 요구하는 서비스 이기 때문에 HTTP 요청으로만 보내기에는 이메일과 비밀번호가 유출이 될수 있는 위험이 높습니다.  비밀번호같은 민감한 정보가 있기때문에 HTTPS를 통해 정보를 암호화 하는 과정이 필요하다고 생각했습니다. HTTPS에서 비대칭키 암호화를 시키기 때문에 사용자의 정보가 아무리 중간에 빼돌려진다해도 절대 복호화가 불가능하기때문에 유출 위험이 현저히 적어집니다. 그래서 자물쇠가 걸려있는걸 보고 사용자들이 마음 놓고 서비스를 이용할수있습니다.
